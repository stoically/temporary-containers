WARNING: New Tabs (about:newtab and about:blank) can make network requests and set cookies, especially when you use the address bar for search engines. If you select "Don't reopen New Tabs in Temporary Containers" here, cookies can get written into and read from the permanent default container as long as the new tab didn't get reopened in a Temporary Container.

If you have a Cookie-Deletion-Add-on that automatically keeps your default/permanent containers clean and you use privacy-oriented search-engines like Startpage.com or DuckDuckGo then it should be no problem to use the "Don't reopen New Tabs" preference.

WARNING: EVERY WEBSITE URL that you visit in a "Deletes History Temporary Container" WILL GET DELETED FROM YOUR ENTIRE HISTORY. This means if you visited the same Website URL in another Container, Temporary Container or in No Container before or while visiting it in a "Deletes History Temporary Container" then those visits will get deleted from History too. This is true until Firefox supports a special History for Container Tabs. The related Firefox bug can be found here.

Be careful. You have been warned. "Deletes History Temporary Containers" tabs have a "-deletes-history" suffix in the container name to remind you.

You can open "Deletes History Temporary Containers" - also with the keyboard shortcut Alt+P - after you read the WARNING and ticked the checkbox.

The deletion applies to the full Website URL, not only the domain. That means, if you e.g. open a specific reddit post or a news article on your favorite news site in a "Deletes History Temporary Container" it won't delete all your previous visits to other reddit posts or news articles that you made outside of "Deletes History Temporary Containers" because the full Website URLs are different.

"Deletes History Temporary Containers" will delete history when the "Deletes History Temporary Container" itself gets deleted after the last Tab in it closes.

After you gave the "Access browsing history" permission you'll be asked to give the permission again with the next update of the Add-on (not again after that). That's a Firefox bug and it's reported here.
